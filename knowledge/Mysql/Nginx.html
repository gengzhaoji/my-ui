深入理解 Nginx 的 root 和 alias 指令 #

Nginx 是一款高性能的 HTTP 和反向代理服务器，也是 IMAP/POP3/SMTP 代理服务器。在 Web 服务器配置中，Nginx 的root和alias指令扮演着至关重要的角色，它们用于指定 Nginx 处理请求时查找文件的根目录或别名目录。

一、Nginx 的 root 指令 #

  1. 基本概念

root指令用于指定请求的根目录。当 Nginx 接收到一个请求时，它会将请求的 URI 附加到root指令指定的路径上，以形成完整的文件路径，并尝试在该路径下查找文件。

  1. 语法与配置段
  • 语法：root path;
  • 默认值：root html;（通常指向 Nginx 安装目录下的 html 文件夹）
  • 配置段：http、server、location、if
  1. 使用示例

假设 Nginx 的配置如下：

server {
    listen 80;
    server_name example.com;
    root /var/www/example;

    location / {
        index index.html;
    }

    location /images {
        root /data;
    }
}
  • 当访问http://example.com/index.html时，Nginx 会尝试在/var/www/example/index.html查找文件。
  • 当访问http://example.com/images/logo.png时，由于/images位置块中单独指定了root /data;，Nginx 会尝试在/data/images/logo.png查找文件，而不是/var/www/example/images/logo.png。
  1. 注意事项
  • root指令的路径末尾是否加斜杠/，对最终的文件路径没有影响。但在配置时，为了清晰和一致性，建议保持路径风格的一致性。
  • 当在location块中指定root时，该root会覆盖外层server或http块中的root设置（如果有的话）。

二、Nginx 的 alias 指令 #

  1. 基本概念

与root不同，alias指令用于指定一个目录别名，它会替换掉 location 匹配到的 URI 部分。换句话说，当 Nginx 处理请求时，它会将 location 匹配到的 URI 替换为alias指定的路径，然后在这个路径下查找文件。

  1. 语法与配置段
  • 语法：alias path;
  • 配置段：仅location
  1. 使用示例

假设 Nginx 的配置如下：

server {
    listen 80;
    server_name example.com;

    location /static {
        alias /var/www/static/;
    }
}

当访问http://example.com/static/css/style.css时，Nginx 会将/static替换为/var/www/static/，然后尝试在/var/www/static/css/style.css查找文件。

  1. 注意事项
  • alias指令的路径末尾必须加斜杠/，以表示这是一个目录。如果不加斜杠，Nginx 可能会将请求当作文件来处理，导致找不到文件。
  • alias不能用于http或server块中，只能在location块中使用。
  • 当使用alias时，location 匹配到的 URI 部分会被完全替换，因此配置时需要特别注意路径的对应关系。

三、root 与 alias 的区别 #

指令作用配置段路径拼接方式路径末尾斜杠要求
root指定请求的根目录http、server、location、ifroot 路径 + location 匹配路径可有可无
alias指定目录别名，替换 location 匹配路径仅 locationalias 路径替换 location 匹配路径必须加

四、总结 #

root和alias是 Nginx 配置中用于指定文件路径的两个重要指令。它们的主要区别在于路径的拼接方式和配置段的不同。理解并正确使用这两个指令，对于高效配置 Nginx 服务器至关重要。在配置时，需要注意路径末尾斜杠的要求，以及路径的对应关系，以确保 Nginx 能够正确地找到并处理请求的文件。

启动 web 服务 (vue 项目为例) #

server {
  # 项目启动端口
  listen            80;
  # 域名（localhost）
  server_name       _;
  # 禁止 iframe 嵌套
  add_header        X-Frame-Options SAMEORIGIN;

  # 访问地址 根路径配置
  location / {
    # 项目目录
    root 	    html;
    # 默认读取文件
    index           index.html;
    # 配置 history 模式的刷新空白
    try_files       $uri $uri/ /index.html;
  }

  # 后缀匹配，解决静态资源找不到问题
  location ~* \.(gif|jpg|jpeg|png|css|js|ico)$ {
    root           html/static/;
  }

  # 图片防盗链
  location ~/static/.*\.(jpg|jpeg|png|gif|webp)$ {
    root              html;
    valid_referers    *.deeruby.com;
    if ($invalid_referer) {
      return          403;
    }
  }

  # 访问限制
  location /static {
    root               html;
    # allow 允许
    allow              39.xxx.xxx.xxx;
    # deny  拒绝
    deny               all;
  }
}

PC 端和移动端使用不同的项目文件映射 #

server {
  location / {
    root /home/static/pc;
    if ($http_user_agent ~* '(mobile|android|iphone|ipad|phone)') {
      root /home/static/mobile;
    }
    index index.html;
  }
}

一个 web 服务，配置多个项目 (location 匹配路由区别) #

server {
  listen                80;
  server_name           _;

  # 主应用
  location / {
    root                html/main;
    index               index.html;
    try_files           $uri $uri/ /index.html;
  }

  # 子应用一
  location ^~ /store/ {
    proxy_pass          http://localhost:8001;
    proxy_redirect      off;
    proxy_set_header    Host $host;
    proxy_set_header    X-Real-IP $remote_addr;
    proxy_set_header    X-Forwarded-For
    proxy_set_header    X-Forwarded-For $proxy_add_x_forwarded_for;
  }

  # 子应用二
  location ^~ /school/ {
    proxy_pass          http://localhost:8002;
    proxy_redirect      off;
    proxy_set_header    Host $host;
    proxy_set_header    X-Real-IP $remote_addr;
    proxy_set_header    X-Forwarded-For $proxy_add_x_forwarded_for;
  }

  # 静态资源读取不到问题处理
  rewrite ^/api/profile/(.*)$ /(替换成正确路径的文件的上一层目录)/$1 last;
}

# 子应用一服务
server {
  listen                8001;
  server_name           _;
  location / {
    root                html/store;
    index               index.html;
    try_files           $uri $uri/ /index.html;
  }

  location ^~ /store/ {
    alias               html/store/;
    index               index.html index.htm;
    try_files           $uri /store/index.html;
  }

  # 接口代理
  location  /api {
    proxy_pass          http://localhost:8089;
  }
}

# 子应用二服务
server {
  listen                8002;
  server_name           _;
  location / {
    root                html/school;
    index               index.html;
    try_files           $uri $uri/ /index.html;
  }

  location ^~ /school/ {
    alias               html/school/;
    index               index.html index.htm;
    try_files           $uri /school/index.html;
  }

  # 接口代理
  location  /api {
    proxy_pass          http://localhost:10010;
  }
}

upstream my_upstream {

  server                http://localhost:9002;
  server                http://localhost:9003;
}

server {
  listen                9000;

  }
}

/prod-api/ 与 /prod-api 的区别详解 #

在 Nginx 中，location 块用于定义请求的路由规则，而 proxy_pass 指令用于将请求代理到后端服务。以下两种配置看似相似，但在实际代理行为中有着显著差异：

配置对比 #

  1. 配置一（末尾带斜杠 /）：

    location /prod-api/ {
    proxy_pass http://127.0.0.1:9400/;
    proxy_redirect off;
}

  2. 配置二（末尾不带斜杠 /）：

    location /prod-api {
    proxy_pass http://127.0.0.1:9400;
    proxy_redirect off;
}

核心区别分析 #

1. location 匹配规则 #

  • /prod-api/
    仅匹配以 /prod-api/ 开头且包含后续路径的请求。例如：

    • /prod-api/user → 匹配成功。
    • /prod-api → 匹配失败（缺少末尾的 /）。
    • /prod-api-test → 匹配失败（非精确前缀）。

  • /prod-api
    匹配以 /prod-api 开头的任何请求，包括：

    • /prod-api/user → 匹配成功。
    • /prod-api → 匹配成功。
    • /prod-api-test → 匹配成功（因为前缀相同）。

2. proxy_pass 路径替换行为 #

  • 配置一（带斜杠）
    proxy_pass 末尾的 / 表示将 移除 location 匹配的路径部分，再将剩余路径拼接到代理地址。
    示例：

    • 请求路径：/prod-api/user/login
    • 代理后的路径：http://127.0.0.1:9400/user/login

  • 配置二（不带斜杠）
    proxy_pass 末尾没有 /，表示将 保留 location 匹配的路径部分，直接拼接到代理地址。
    示例：

    • 请求路径：/prod-api/user/login
    • 代理后的路径：http://127.0.0.1:9400/prod-api/user/login

对比表格 #

行为配置一（/prod-api/）配置二（/prod-api）
匹配路径严格匹配 /prod-api/ 开头的请求匹配所有以 /prod-api 开头的请求
代理路径处理移除 /prod-api/ 前缀保留 /prod-api 前缀
请求 /prod-api不匹配代理到 http://127.0.0.1:9400/prod-api
请求 /prod-api/代理到 http://127.0.0.1:9400/代理到 http://127.0.0.1:9400/prod-api/
请求 /prod-api/user代理到 http://127.0.0.1:9400/user代理到 http://127.0.0.1:9400/prod-api/user

适用场景 #

  1. 配置一（/prod-api/）

    • 后端服务不需要 prod-api 前缀（如微服务网关或统一 API 入口）。
    • 需要隐藏路径前缀，实现路径“清洗”。
    • 示例：将前端请求的 /prod-api/user 代理到后端的 /user。

  2. 配置二（/prod-api）

    • 后端服务需要保留 prod-api 前缀（如未做路径隔离的旧服务）。
    • 需要精确传递完整路径的场景。
    • 示例：代理到另一个服务，且该服务通过 /prod-api 区分接口版本。

常见问题 #

1. 404 错误 #

  • 原因：配置一未正确匹配 /prod-api（不带斜杠）的请求，导致直接访问 /prod-api 时返回 404。
  • 解决：添加 location = /prod-api { ... } 处理根路径。

2. 路径冗余 #

  • 现象：配置二导致后端收到带有 /prod-api 前缀的请求，但后端未配置该路径。
  • 解决：改用配置一或在后端服务中添加前缀处理逻辑。

总结 #

  • proxy_pass 末尾的斜杠决定了是否保留 location 匹配的路径。
  • location 的精确性影响请求的匹配范围。
  • 根据后端服务的路径需求选择配置，避免路径不匹配导致的异常。
